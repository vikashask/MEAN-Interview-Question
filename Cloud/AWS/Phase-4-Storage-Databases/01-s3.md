# S3 (Simple Storage Service)

> **In plain English:** S3 is AWS's file storage — you throw files (called "objects") into buckets and AWS keeps them safe, durable, and reachable from anywhere, at basically infinite scale. There's no traditional folder structure under the hood — folder-looking paths are just part of the object's name (key).

## Real-world analogy

Think of S3 as a massive, infinitely expandable self-storage warehouse:

- **Bucket** = one storage unit with a globally unique name (no two customers in the world can have the same bucket name).
- **Object** = one item stored inside (a file), identified by its **Key** (its full "path/name", e.g. `photos/2024/cat.jpg` — even though it looks like folders, S3 doesn't really have folders, just names with slashes in them).
- **Storage Class** = which shelf you put the item on: Standard = front shelf, instantly grabbable, priciest. Glacier = a deep basement vault, cheapest, but takes time to retrieve.
- **Versioning** = keeping every previous edition of an item instead of overwriting it, so you can always go back.
- **Bucket Policy** = the sign on the storage unit's door saying who's allowed in and what they can do.
- **Lifecycle Policy** = an automatic "move to cheaper storage after X days, delete after Y days" janitor.
- **Presigned URL** = a temporary, time-limited key you can hand to a friend so they can grab (or drop off) one specific item without having their own storage unit key.

## Core concepts (memorize these first)

| Term | What it means |
|---|---|
| **Bucket** | Top-level container, globally unique name across all of AWS. |
| **Object** | The actual file, identified by a **Key** (its full name/path). |
| **Storage Class** | Standard (frequent access) → Standard-IA (infrequent) → Glacier (archive, slow retrieval) → Deep Archive (cheapest, hours to retrieve). |
| **Bucket Policy** | JSON access rules attached to the bucket itself (who from outside can access it). |
| **ACL** | Older, more limited per-object/bucket permission system — mostly replaced by bucket policies + IAM. |
| **Versioning** | Keeps every version of an object ever uploaded; a delete just adds a "delete marker" instead of truly erasing (recoverable). |
| **Lifecycle Policy** | Rules to auto-transition objects to cheaper storage classes or auto-delete them after N days. |
| **Multipart Upload** | Splits a large file into parts uploaded in parallel — required for files >5GB, recommended for anything large. |
| **Presigned URL** | A temporary signed link granting time-limited access to a private object, without making the object itself public. |
| **Static Website Hosting** | S3 can directly serve HTML/CSS/JS as a website — no server needed. |
| **Cross-Region Replication (CRR)** | Auto-copies objects to a bucket in another region — used for disaster recovery / lower latency for distant users. |
| **S3 Event Notifications** | Fires an event (to Lambda, SQS, or SNS) automatically whenever an object is created/deleted — the backbone of many "upload triggers processing" pipelines. |

## Memory hooks

- **"Object = file + Key = its full name."** There's no real folder — `images/cat.jpg` is just one long name with a slash in it.
- Storage class ladder (cheap ↔ slow to retrieve): **Standard → Standard-IA → One Zone-IA → Glacier → Glacier Deep Archive.** Going down the ladder = cheaper storage, slower/costlier retrieval.
- **Presigned URL = a temporary house key you hand someone, they don't need their own key (IAM credentials) to get in.**
- Once versioning is turned ON, it can only be *suspended*, never fully turned back OFF.

## Basic S3 Operations

The everyday create/list/upload/download/sync/delete commands.

```bash
# Create bucket
aws s3 mb s3://my-bucket

# List buckets
aws s3 ls

# List objects in bucket
aws s3 ls s3://my-bucket
aws s3 ls s3://my-bucket/folder/ --recursive

# Upload file
aws s3 cp file.txt s3://my-bucket/
aws s3 cp file.txt s3://my-bucket/ --storage-class INTELLIGENT_TIERING

# Download file
aws s3 cp s3://my-bucket/file.txt ./

# Sync directory
aws s3 sync ./local-folder s3://my-bucket/remote-folder/
aws s3 sync s3://my-bucket/remote-folder/ ./local-folder

# Delete file
aws s3 rm s3://my-bucket/file.txt

# Delete bucket
aws s3 rb s3://my-bucket --force
```

## S3 with SDK

Note the `Upload` helper class specifically — it automatically handles multipart upload and progress tracking for large files, instead of you managing parts manually.

```javascript
import {
  S3Client,
  PutObjectCommand,
  GetObjectCommand,
  ListObjectsV2Command,
  DeleteObjectCommand,
} from "@aws-sdk/client-s3";
import { getSignedUrl } from "@aws-sdk/s3-request-presigner";
import { Upload } from "@aws-sdk/lib-storage";
import fs from "fs";

const s3 = new S3Client({ region: "us-east-1" });

// Upload file
const uploadFile = async (bucket, key, filePath) => {
  const fileStream = fs.createReadStream(filePath);
  const command = new PutObjectCommand({
    Bucket: bucket,
    Key: key,
    Body: fileStream,
    ContentType: "application/pdf",
    Metadata: {
      uploadedBy: "user123",
      uploadDate: new Date().toISOString(),
    },
  });

  await s3.send(command);
};

// Upload large file with multipart
const uploadLargeFile = async (bucket, key, filePath) => {
  const fileStream = fs.createReadStream(filePath);
  const upload = new Upload({
    client: s3,
    params: {
      Bucket: bucket,
      Key: key,
      Body: fileStream,
    },
    queueSize: 4,
    partSize: 5 * 1024 * 1024, // 5MB parts
  });

  upload.on("httpUploadProgress", (progress) => {
    console.log(`Uploaded ${progress.loaded} of ${progress.total} bytes`);
  });

  await upload.done();
};

// Download file
const downloadFile = async (bucket, key) => {
  const command = new GetObjectCommand({
    Bucket: bucket,
    Key: key,
  });

  const response = await s3.send(command);
  const content = await response.Body.transformToString();
  return content;
};

// List objects
const listObjects = async (bucket, prefix = "") => {
  const command = new ListObjectsV2Command({
    Bucket: bucket,
    Prefix: prefix,
    MaxKeys: 1000,
  });

  const response = await s3.send(command);
  return response.Contents;
};

// Delete object
const deleteObject = async (bucket, key) => {
  const command = new DeleteObjectCommand({
    Bucket: bucket,
    Key: key,
  });

  await s3.send(command);
};

// Generate presigned URL
const getPresignedUrl = async (bucket, key, expiresIn = 3600) => {
  const command = new GetObjectCommand({
    Bucket: bucket,
    Key: key,
  });

  const url = await getSignedUrl(s3, command, { expiresIn });
  return url;
};
```

## Bucket Policies

The door sign — who from outside the bucket's owner is allowed to do what. Common patterns: public read for static assets, IP-restricted access, CloudFront-only access, or cross-account sharing.

```json
// Public read access
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadGetObject",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}

// Restrict to specific IP
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "203.0.113.0/24"
        }
      }
    }
  ]
}

// CloudFront access only
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::cloudfront:user/CloudFront Origin Access Identity ABC123"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}

// Cross-account access
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:root"
      },
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

## Lifecycle Policies

The automatic janitor: move objects to cheaper storage as they age, clean up incomplete uploads, and expire old data — all without you lifting a finger.

```json
{
  "Rules": [
    {
      "Id": "MoveToIA",
      "Status": "Enabled",
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 90,
          "StorageClass": "GLACIER"
        },
        {
          "Days": 365,
          "StorageClass": "DEEP_ARCHIVE"
        }
      ]
    },
    {
      "Id": "DeleteOldVersions",
      "Status": "Enabled",
      "NoncurrentVersionExpiration": {
        "NoncurrentDays": 30
      }
    },
    {
      "Id": "DeleteIncompleteUploads",
      "Status": "Enabled",
      "AbortIncompleteMultipartUpload": {
        "DaysAfterInitiation": 7
      }
    },
    {
      "Id": "DeleteTempFiles",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "temp/"
      },
      "Expiration": {
        "Days": 7
      }
    }
  ]
}
```

## S3 Versioning

Once enabled, every overwrite/delete keeps the old copy around instead of losing it — protects against accidental overwrites or deletes.

```bash
# Enable versioning
aws s3api put-bucket-versioning \
  --bucket my-bucket \
  --versioning-configuration Status=Enabled

# List versions
aws s3api list-object-versions --bucket my-bucket

# Get specific version
aws s3api get-object \
  --bucket my-bucket \
  --key file.txt \
  --version-id version-id \
  output.txt

# Delete specific version
aws s3api delete-object \
  --bucket my-bucket \
  --key file.txt \
  --version-id version-id
```

## S3 Replication

Cross-Region Replication (CRR) automatically copies objects to a bucket in another region — useful for disaster recovery, compliance (data residency), or reducing latency for users far from the primary region.

```json
// Cross-region replication configuration
{
  "Role": "arn:aws:iam::123456789012:role/s3-replication-role",
  "Rules": [
    {
      "Status": "Enabled",
      "Priority": 1,
      "DeleteMarkerReplication": {
        "Status": "Enabled"
      },
      "Filter": {
        "Prefix": "documents/"
      },
      "Destination": {
        "Bucket": "arn:aws:s3:::my-replica-bucket",
        "ReplicationTime": {
          "Status": "Enabled",
          "Time": {
            "Minutes": 15
          }
        },
        "Metrics": {
          "Status": "Enabled",
          "EventThreshold": {
            "Minutes": 15
          }
        }
      }
    }
  ]
}
```

## S3 Event Notifications

Fire an event automatically whenever something happens to an object (uploaded, deleted) — the classic pattern is "user uploads image → Lambda auto-generates a thumbnail."

```json
// Lambda notification configuration
{
  "LambdaFunctionConfigurations": [
    {
      "LambdaFunctionArn": "arn:aws:lambda:us-east-1:123456789012:function:ProcessImage",
      "Events": ["s3:ObjectCreated:*"],
      "Filter": {
        "Key": {
          "FilterRules": [
            {
              "Name": "prefix",
              "Value": "images/"
            },
            {
              "Name": "suffix",
              "Value": ".jpg"
            }
          ]
        }
      }
    }
  ],
  "QueueConfigurations": [
    {
      "QueueArn": "arn:aws:sqs:us-east-1:123456789012:my-queue",
      "Events": ["s3:ObjectRemoved:*"]
    }
  ]
}
```

## S3 Static Website Hosting

S3 can serve a whole static website (HTML/CSS/JS) directly, no web server needed. In production this is almost always fronted by CloudFront for HTTPS + caching.

```bash
# Enable website hosting
aws s3 website s3://my-bucket \
  --index-document index.html \
  --error-document error.html

# Upload website files
aws s3 sync ./dist s3://my-bucket --delete

# Set public read policy
aws s3api put-bucket-policy \
  --bucket my-bucket \
  --policy file://public-policy.json
```

## S3 Performance Optimization

S3 scales automatically, but two habits help at high request rates: run uploads in parallel, and avoid giving many objects a shared predictable prefix (older S3 partitioning was sensitive to this — modern S3 handles it much better, but it's still a common interview topic).

```javascript
// Parallel uploads for better performance
import { Upload } from "@aws-sdk/lib-storage";

const parallelUpload = async (bucket, files) => {
  const uploads = files.map((file) => {
    const upload = new Upload({
      client: s3,
      params: {
        Bucket: bucket,
        Key: file.name,
        Body: file.stream,
      },
      queueSize: 4,
      partSize: 10 * 1024 * 1024, // 10MB parts
    });

    return upload.done();
  });

  await Promise.all(uploads);
};

// Prefix distribution for high request rates
// Instead of: user123/file1.txt, user123/file2.txt
// Use:        a/user123/file1.txt, b/user123/file2.txt
const generateKey = (userId, fileName) => {
  const hash = userId.charCodeAt(0) % 10;
  return `${hash}/${userId}/${fileName}`;
};
```

## S3 Select

Lets you run a SQL-like query directly on an object stored in S3 (e.g. a CSV) and get back only the matching rows — instead of downloading the whole file and filtering client-side.

```javascript
import { SelectObjectContentCommand } from "@aws-sdk/client-s3";

// Query CSV file
const queryCSV = async (bucket, key) => {
  const command = new SelectObjectContentCommand({
    Bucket: bucket,
    Key: key,
    ExpressionType: "SQL",
    Expression: "SELECT * FROM s3object s WHERE s.age > 25",
    InputSerialization: {
      CSV: {
        FileHeaderInfo: "USE",
        RecordDelimiter: "\n",
        FieldDelimiter: ",",
      },
    },
    OutputSerialization: {
      JSON: {
        RecordDelimiter: "\n",
      },
    },
  });

  const response = await s3.send(command);

  let result = "";
  for await (const event of response.Payload) {
    if (event.Records) {
      result += event.Records.Payload.toString();
    }
  }

  return result;
};
```

## Quick interview answers

**Q: Does S3 have real folders?**
No. Everything is a flat list of objects; a "folder" is just a common prefix in the object key (e.g. `images/cat.jpg`). The console displays them as folders for convenience only.

**Q: What's the difference between the storage classes?**
They trade cost for retrieval speed/availability: Standard (frequent access, instant), Standard-IA/One Zone-IA (infrequent access, instant but cheaper, retrieval fee), Glacier (archive, minutes-to-hours retrieval), Deep Archive (cheapest, up to 12 hours retrieval).

**Q: How do you protect against accidental object deletion?**
Enable Versioning (deletes become recoverable "delete markers") and optionally MFA Delete (requires MFA to permanently delete a version).

**Q: How does a presigned URL work without exposing your AWS credentials?**
Your backend (which has real credentials) signs a URL with a time-limited signature; anyone holding that URL can perform the one allowed action (e.g. GET) until it expires, without ever seeing your actual keys.

**Q: How would you make S3 uploads trigger processing automatically?**
S3 Event Notifications — configure the bucket to invoke a Lambda function (or push to SQS/SNS) on `ObjectCreated` events.

**Q: Multipart upload — when is it required vs recommended?**
Required for objects larger than 5GB. Recommended for anything roughly over 100MB, since it uploads parts in parallel and lets you resume failed parts instead of restarting the whole file.
